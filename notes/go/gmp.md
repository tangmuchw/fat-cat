# Go Runtime 核心深度解析

精准对应**GMP调度、GC机制、Channel原理、高并发竞态（Race Condition）、OOM排查**五大核心点，覆盖原理+实战+问题解决方案，完全匹配安全/后端高并发场景需求。

## 一、GMP 调度模型（Go 高并发核心）

### 1. 核心概念

Go 摒弃了内核线程调度，自研**用户态协程调度**，三大核心组件：

- **G（Goroutine）**：用户态轻量级协程，栈初始2KB，开销极小，是业务执行单元
- **M（Machine）**：内核线程，由操作系统调度，真正执行代码的载体
- **P（Processor）**：逻辑处理器，**持有G队列、上下文、CPU核心**，决定并发度（`GOMAXPROCS`）

### 2. 调度流程（核心机制）

1. 全局队列（Global Queue）：存放等待运行的 G
2. P 的本地队列：私有队列，无锁访问，优先执行
3. M 必须绑定 P 才能运行 G，M 从 P 本地队列取 G 执行
4. 本地队列空 → 偷取其他 P 的 G（**工作窃取算法**）→ 从全局队列取 G
5. 系统调用/阻塞时：M 释放 P，P 绑定其他空闲 M，保证 CPU 利用率

### 3. 高并发实战关键点

- **阻塞处理**：
    - 用户态阻塞（`channel`、`sync`）：**不释放M/P**，调度器切换其他G
    - 系统调用阻塞（文件IO、网络IO）：**解绑P**，P 继续调度其他G，无CPU浪费
- **常见问题**：
    - `GOMAXPROCS` 设置过小 → CPU 利用率低
    - 大量G阻塞在系统调用 → M 暴涨，导致线程泄漏
    - 全局队列加锁竞争 → 调度延迟

## 二、Go 垃圾回收（GC）机制（内存安全 + 低延迟）

### 1. 核心特性

Go 1.5+ 采用**三色标记法 + 混合写屏障**，实现**并发GC**，STW（Stop The World）极低（微秒级）。

### 2. 三色标记流程

1. **白色**：未扫描对象
2. **灰色**：已扫描，引用对象未扫描
3. **黑色**：已扫描，无白色引用
4. 写屏障：并发GC时，记录对象引用变更，**避免漏标**

### 3. GC 触发条件

- 内存阈值：默认堆内存增长**100%**触发
- 定时触发：2分钟无GC则强制触发
- 手动触发：`runtime.GC()`

### 4. 高并发 OOM 实战解决方案

1. **内存泄漏**：
    - 长生命周期对象持有短生命周期对象（如全局map、goroutine泄漏）
    - 切片未释放底层数组（`slice = nil` 而非只截断）
2. **GC 压力过大**：
    - 大量临时小对象 → 改用**sync.Pool**复用对象
    - 大内存分配 → 避免频繁创建销毁
3. **排查工具**：
    - `pprof` 内存分析、goroutine 分析
    - `GODEBUG=gctrace=1` 监控GC频率、耗时

## 三、Channel 原理（Go 并发通信基石）

### 1. 核心本质

**通信共享内存，而非内存共享通信**，底层是**环形队列 + 锁 + 等待队列**。

### 2. 底层结构

- 无缓冲channel：同步通信，发送方必须等待接收方
- 有缓冲channel：异步通信，缓冲区满则阻塞发送，空则阻塞接收
- 等待队列：保存阻塞的 G（sendq/recvq）

### 3. 关键特性

- 关闭channel后，发送panic，接收返回零值
- nil channel 永久阻塞
- 用于协程同步、限流、消息传递

### 4. 高并发坑点

- 未关闭channel → goroutine泄漏
- 重复关闭/向关闭channel发送 → panic
- 大量channel阻塞 → 调度延迟

## 四、Race Condition（竞态条件）实战处理

### 1. 定义

多个goroutine**同时读写共享变量**，无同步机制，导致结果不可预期。

### 2. 检测工具

```bash
# 自动检测竞态（开发必备）
go run -race main.go
go test -race ./...
```

### 3. 解决方案

1. **channel 同步**：推荐，符合Go哲学
2. **sync.Mutex/RWMutex**：读写锁，读多写少场景用RWMutex
3. **atomic 原子操作**：简单数值操作（int64、bool），性能最高
4. **sync.Once**：单例初始化，避免重复执行

## 五、OOM 问题实战排查流程

### 1. 快速定位

1. 查看进程内存占用：`top`、`free`
2. 开启GC日志：`export GODEBUG=gctrace=1`
3. pprof 抓取内存快照：
    ```go
    _ "net/http/pprof"
    // 访问：http://ip:port/debug/pprof/heap
    ```

### 2. 高频OOM原因

1. **goroutine 泄漏**：无限创建协程，无退出机制
2. **全局map无限增长**：未清理过期数据
3. **大对象未释放**：长生命周期引用大切片/数组
4. **GC 被阻塞**：cgo 阻塞、CPU 满载

### 3. 根治方案

- 协程：用**协程池**控制数量，禁止无限制创建
- 内存：用`sync.Pool`复用小对象，定期清理全局容器
- 监控：接入pprof、监控goroutine数、堆内存大小
