# 一、乐观锁

## 1. 核心概念

**乐观锁**：默认**没人会改数据、冲突很少**，不上锁，只在**提交更新时**检查数据有没有被别人改过。

- 心态：乐观，假设并发冲突概率低
- 特点：不加排他锁，并发性能高

## 2. 实现方式（常用两种）

1. **版本号机制**
   表加 `version` 字段，每次更新版本+1

```sql
-- 更新时校验版本
update user set name='xx',version=version+1
where id=1 and version=1;
```

别人先改完，version 变了，本条更新失败。

2. **CAS 字段比对**
   更新时对比旧数据，只在原值不变时才更新

```sql
update stock set num=num-1
where id=1 and num=100;
```

## 3. 适用场景

读多写少、并发冲突少的业务（商品详情、用户信息、文章）

# 二、悲观锁

和乐观锁相反：
**悲观锁**：默认**一定会有人抢改、冲突多**，操作前直接加锁，锁住资源，别人不能改，用完再释放。

- 心态：悲观，默认并发争抢严重
- 特点：强一致性，并发性能差

## 常见实现

1. **数据库行锁/表锁**

```sql
-- 悲观行锁，查询即加锁
select * from user where id=1 for update;
```

2. 分布式锁：Redis、Zookeeper 锁

适用：写多、强一致性要求高（订单扣款、库存扣减、资金交易）

# 三、常见锁分类（最全梳理）

## 1. 按并发思想分

- 乐观锁：版本号、CAS
- 悲观锁：排他锁、for update、分布式锁

## 2. 按锁粒度分

- **表锁**：锁住整张表，开销大、并发极低
- **行锁**：只锁一行，粒度细、并发高（InnoDB 默认）
- **页锁**：介于表锁&行锁之间

## 3. 按锁类型分

- **共享锁（S锁/读锁）**：
  多人可以同时读，不能改
- **排他锁（X锁/写锁）**：
  只能一人读写，其他人全部阻塞

## 4. 开发高频其他锁

- **分布式锁**：多服务/多实例争抢资源时用（Redis Redlock、Zk）
- **读写锁**：读共享、写排他，适合读多写少
- **自旋锁**：循环等待抢锁，不阻塞休眠，适合短任务
- **偏向锁/轻量级锁/重量级锁**：Java JVM 锁升级

# 四、快速对比总结

| 类型   | 加锁时机   | 性能 | 一致性   | 场景           |
| ------ | ---------- | ---- | -------- | -------------- |
| 乐观锁 | 更新时校验 | 高   | 最终一致 | 读多写少       |
| 悲观锁 | 操作前加锁 | 低   | 强一致   | 写多、交易库存 |
